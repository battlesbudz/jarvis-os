# Production Context

## Purpose
This is the execution room for turning an approved idea, plan, script, or research result into a concrete deliverable. Production should reduce ambiguity before building.

## Route Here When
- The user wants to make, build, package, ship, animate, design, or assemble something.
- A content script needs to become a video/asset/workflow.
- A business plan needs a brief, spec, SOP, or packaged output.
- Jarvis needs to translate strategy into production steps.

## Read First
1. Active source artifact from `content-studio/`, `business/`, or `research/`.
2. Existing matching brief in `briefs/`.
3. Existing matching spec in `specs/`.
4. Related build notes in `builds/` only when continuing work.

## Prefer
- Brief before spec.
- Spec before build.
- Clear acceptance criteria.
- Explicit owner, tools, files, and output destination.
- Small testable deliverables over giant undefined builds.

## Skip Unless Needed
- Full content history when a final script or source artifact exists.
- Research folders unless the production output makes factual claims.
- Personal-life context unless the deliverable is personal.
- App source code unless the production task is implementation.

## Pipeline
1. Brief: define audience, goal, source, constraints, and success criteria.
2. Spec: define structure, steps, tools, assets, risks, and acceptance checks.
3. Build: create the deliverable or implementation notes.
4. Output: package final files, summary, and next actions.

## Output Types
- Production briefs
- Technical or creative specs
- Build notes
- Asset lists
- Acceptance checklists
- Packaged outputs

## Approval Boundaries
Do not publish, deploy, spend money on tools/assets, or trigger external production systems without explicit approval.

## Handoff Back To PRIME
Return the production stage, created/updated file, unresolved inputs, and the next build action.
